# 게시 흐름

>[!NOTE]
>
>Adobe Experience Platform Launch은 Experience Platform에서 데이터 수집 기술 세트로 브랜드 재지정되었습니다. 그 결과 제품 설명서에서 몇 가지 용어 변경 사항이 롤아웃되었습니다. 용어 변경 내용을 통합 참조하려면 다음 [document](../../term-updates.md)을 참조하십시오.

Adobe Experience Platform에서 태그 게시 흐름은 라이브러리를 만들고, 빌드를 테스트하고, 프로덕션에 대해 승인하는 프로세스를 말합니다.

라이브러리에서 수행할 수 있는 사용 가능한 작업은 라이브러리의 상태와 사용자의 권한 수준에 따라 다릅니다. 또한 라이브러리의 상태는 게시 플로우의 업스트림에 따라 포함 리소스(규칙, 데이터 요소 및 확장)에도 영향을 줍니다.

아래 섹션에서는 게시 흐름과 관련된 권한, 라이브러리 상태 및 업스트림에 대한 세부 사항을 다룹니다.

## 권한 {#permissions}

게시 흐름에 중요한 사용자 권한 수준에는 여러 가지가 있습니다. 특히 [!UICONTROL 개발], [!UICONTROL 승인] 및 [!UICONTROL 게시] 속성 권한:

* **[!UICONTROL 개발]**: 라이브러리를 만들고 개발용으로 빌드하고 승인을 위해 제출할 권한이 포함되어 있습니다.
* **[!UICONTROL 승인]**: 스테이징을 위해 빌드하고 스테이징된 빌드를 승인할 권한이 포함되어 있습니다.
* **[!UICONTROL 게시]**: 승인된 라이브러리를 게시할 권한이 포함되어 있습니다.

이러한 권한은 포괄적이지 않습니다. 한 사람이 처음부터 끝까지 작업 과정을 수행하려면 주어진 속성 내에 있는 세 개의 권한을 모두 부여받아야 합니다.

태그의 권한 관리에 대한 자세한 내용은 [사용 권한 안내서](../administration/user-permissions.md)를 참조하십시오.

## 라이브러리 상태 {#state}

게시 흐름과 관련하여 라이브러리가 있을 수 있는 네 가지 기본 상태가 있습니다.

* [[!UICONTROL 개발]](#development)
* [[!UICONTROL 제출됨]](#submitted)
* [[!UICONTROL 승인됨]](#approved)
* [[!UICONTROL 게시됨]](#published)

이 네 가지 상태는 데이터 수집 UI의 **[!UICONTROL 게시 흐름]** 탭 내에 열로 표시되어 있습니다.

![](./images/approval-workflow/flow-ui.png)

이러한 상태 간에 라이브러리를 이동하려면 특정 작업을 수행해야 합니다. 다음 다이어그램은 상태 간에 라이브러리를 이동하는 각 작업에 대해 설명합니다.

![](./images/approval-workflow/library-state.png)

### [!UICONTROL 개발] {#development}

새 라이브러리가 만들어지면 [!UICONTROL 개발] 상태에서 시작합니다. 라이브러리가 [!UICONTROL Development]에 있는 동안 라이브러리를 변경해야 합니다. 개발 및 테스트가 완료되면 라이브러리를 제출하여 승인을 받을 수 있습니다.

다음 표에서는 [!UICONTROL 개발] 상태의 라이브러리에 사용할 수 있는 작업에 대해 설명합니다.

| 작업 | 설명 |
| --- | --- |
| [!UICONTROL 편집] | [!UICONTROL 라이브러리 편집] 화면을 사용하여 라이브러리에서 구성 요소를 추가하거나 제거합니다. |
| [!UICONTROL 개발에 빌드] | 라이브러리에 대한 빌드를 만듭니다. 빌드는 컴파일되고 라이브러리가 지정된 환경에 배포됩니다. 라이브러리가 환경에 할당되지 않았거나 업스트림에 이미 정의된 변경 사항이 포함된 경우 이 단계가 실패합니다. |
| [!UICONTROL 승인을 위한 제출] | 개발 환경에서 라이브러리 할당을 취소하고 승인 권한이 있는 사용자의 [!UICONTROL Submitted] 열로 라이브러리를 이동합니다. 이 옵션을 사용하려면 라이브러리에 대한 최신 빌드가 성공해야 합니다. |
| [!UICONTROL 스테이징에 제출 및 빌드] | 이 작업은 개발 및 승인 권한이 모두 있는 사용자만 수행할 수 있습니다. 이 작업은 개발 환경에서 라이브러리 할당을 취소하고, 라이브러리를 [!UICONTROL Submitted] 상태로 이동하고, 라이브러리를 스테이징 환경으로 빌드합니다. 이 옵션을 사용하려면 라이브러리에 대한 최신 빌드가 성공해야 합니다. |
| [!UICONTROL Approve for Publishing] | 이 작업은 개발 및 승인 권한이 모두 있는 사용자만 수행할 수 있습니다. 이 작업은 개발 환경에서 라이브러리 할당을 취소하고 [!UICONTROL Approved] 상태로 이동합니다. 스테이징 환경을 건너뛰고 [!UICONTROL Submitted] 상태를 완전히 건너뜁니다. 이 옵션을 사용하려면 라이브러리에 대한 최신 빌드가 성공해야 합니다. |
| [!UICONTROL 승인 및 프로덕션에 게시] | 이 작업은 개발, 승인 및 게시 권한이 있는 사용자만 수행할 수 있습니다. 이 작업은 개발 환경에서 라이브러리 할당을 취소하고, [!UICONTROL Approved] 상태로 이동한 후 프로덕션에 게시합니다. 프로덕션 빌드가 완료되면 라이브러리가 [!UICONTROL Published] 상태로 이동합니다. 이 옵션을 사용하려면 라이브러리에 대한 최신 빌드가 성공해야 합니다. |
| [!UICONTROL 삭제] | platform launch에서 라이브러리를 제거합니다. 이렇게 해도 환경에서 빌드가 제거되지는 않습니다. |

### [!UICONTROL 제출됨] {#submitted}

라이브러리가 [!UICONTROL Submitted] 상태에 있으면 승인 권한이 있는 사용자가 스테이징 환경에서 라이브러리를 테스트할 수 있습니다. 테스트가 완료되면 라이브러리를 승인하거나 거부할 수 있습니다. 거부된 빌드는 다시 [!UICONTROL 개발]으로 이동되므로 게시 흐름을 다시 시작하기 전에 추가 변경 사항을 수행할 수 있습니다.

다음 표에서는 [!UICONTROL Submitted] 상태의 라이브러리에 사용할 수 있는 작업에 대해 설명합니다.

| 작업 | 설명 |
| --- | --- |
| [!UICONTROL 열기] | 라이브러리의 콘텐츠를 봅니다. [!UICONTROL Development] 열 이외의 라이브러리에는 변경할 수 없습니다. 변경해야 하는 경우 [!UICONTROL Development]에서 변경할 수 있도록 라이브러리를 거부해야 합니다. |
| [!UICONTROL Build for Staging] | 배포를 위한 스테이징 환경에서 라이브러리를 빌드합니다. |
| [!UICONTROL 게시 승인] | 게시 권한이 있는 사용자가 작업할 수 있도록 라이브러리를 [!UICONTROL Approved] 열로 이동합니다. |
| [!UICONTROL 승인 및 프로덕션에 게시] | 이 작업은 승인 및 게시 권한이 모두 있는 사용자만 수행할 수 있습니다. 이 작업은 스테이징 환경에서 라이브러리 할당을 취소하고, [!UICONTROL Approved] 상태로 이동한 후 프로덕션에 게시합니다. 프로덕션 빌드가 완료되면 라이브러리가 [!UICONTROL Published] 상태로 이동합니다. 이 작업은 스테이징 환경에서 성공적으로 빌드하지 않고 를 사용하여 수행할 수 있습니다. |
| [!UICONTROL 거부] | 스테이징 환경에서 라이브러리 할당을 취소하고 다시 [!UICONTROL Development] 열로 라이브러리를 이동합니다. |

### [!UICONTROL 승인됨] {#approved}

라이브러리가 승인되면 게시 권한이 있는 사용자는 라이브러리를 게시하거나 거부할 수 있습니다. 거부된 빌드는 다시 [!UICONTROL 개발]으로 이동되므로 게시 흐름이 다시 시작되기 전에 추가 변경 사항을 적용할 수 있습니다.

다음 표에서는 [!UICONTROL Approved] 상태의 라이브러리에 사용할 수 있는 작업에 대해 설명합니다.

| 작업 | 설명 |
| --- | --- |
| [!UICONTROL 열기] | 라이브러리의 콘텐츠를 봅니다. [!UICONTROL Development] 열 이외의 라이브러리에는 변경할 수 없습니다. 변경해야 하는 경우 [!UICONTROL Development]에서 변경할 수 있도록 라이브러리를 거부해야 합니다. |
| [!UICONTROL 빌드 및 프로덕션에 게시] | 스테이징 환경에서 라이브러리 할당을 취소하고, 프로덕션 환경에 라이브러리를 할당하고, 배포합니다.<br><br>**중요**: 이 옵션을 선택하면 라이브러리가 프로덕션 환경에서 활성화됩니다. 이 옵션을 선택하기 전에 라이브러리에 원하는 변경 사항이 포함되어 있는지 확인합니다. |
| [!UICONTROL 거부] | 스테이징 환경에서 라이브러리 할당을 취소하고 추가 변경을 위해 라이브러리를 [!UICONTROL Development] 열로 이동합니다. |

### [!UICONTROL 게시됨] {#published}

[!UICONTROL Published] 열에는 게시된 라이브러리 및 게시 날짜가 표시됩니다. 현재 게시된 라이브러리 옆에는 녹색 점이 표시됩니다. 이전 라이브러리에 다시 게시하지 않은 경우 항상 열 맨 위에 있는 라이브러리가 됩니다.

| 작업 | 설명 |
| --- | --- |
| [!UICONTROL 열기] | 라이브러리의 콘텐츠를 봅니다. [!UICONTROL Development] 열 이외의 라이브러리에는 변경할 수 없습니다. 프로덕션 환경에서 변경하려는 경우 새 라이브러리를 만들고 전체 게시 프로세스를 통해 이동해야 합니다. |
| [!UICONTROL 다시 게시] | 이 작업은 가장 최근에 게시된 5개의 라이브러리에서만 사용할 수 있으며, 프로덕션 환경이 (A) 아카이브 옵션을 비활성화하여 구성되고 (b) 빌드 시 Adobe] 호스트에서 관리 [!UICONTROL 하는 경우에만 사용할 수 있습니다. |
| [!UICONTROL 다운로드] | 이 작업은 가장 최근에 게시된 5개의 라이브러리에서만 사용할 수 있으며, 프로덕션 환경이 (A)에 아카이브 옵션을 사용하여 구성되고 (b) 빌드 시 Adobe] 호스트에서 관리 [!UICONTROL 하는 경우에만 사용할 수 있습니다. |

## 업스트림 {#upstream}

첫 번째 라이브러리를 게시한 후에는 게시 플로우를 통해 최신 라이브러리를 이동할 때 업스트림 역할을 이해하는 것이 중요합니다.

라이브러리가 현재 [!UICONTROL Development], [!UICONTROL Submitted] 또는 [!UICONTROL Approved] 단계에 있는 경우 해당 라이브러리는 업스트림 라이브러리의 규칙, 데이터 요소 및 확장을 상속합니다. 이러한 상속된 리소스는 게시 플로우를 통해 이동할 때 각 라이브러리에 대한 &quot;기준&quot;을 구성합니다. 기본적으로 업스트림에 의해 설정된 기준선에 대한 일련의 변경 사항으로 각 새 라이브러리를 생각할 수 있습니다. 이렇게 하면 새 이터레이션이 게시될 때 이전 라이브러리에서 예기치 않게 아무것도 덮어쓰지 않습니다.

업스트림에 포함된 내용은 라이브러리의 현재 단계에 따라 다릅니다. 예를 들어 [!UICONTROL Approved] 열의 라이브러리는 [!UICONTROL Published] 라이브러리의 리소스만 상속하지만, [!UICONTROL Development] 아래의 라이브러리는 다른 모든 열의 리소스를 상속합니다.

![](./images/approval-workflow/upstream.png)

데이터 수집 UI에서 라이브러리를 편집하는 경우 업스트림에서 상속된 모든 리소스는 **[!UICONTROL 리소스 업스트림]** 섹션에 표시됩니다. 이러한 리소스를 보려면 섹션 머리글 아래에서 확장 탭을 선택합니다.

![](./images/approval-workflow/upstream-collapse.png)

섹션을 확장하여 업스트림에서 상속된 개별 리소스를 표시합니다. 왼쪽 레일을 사용하여 [!UICONTROL 규칙], [!UICONTROL 데이터 요소] 및 [!UICONTROL 확장] 간을 필터링하거나 검색 막대를 사용하여 이름별로 특정 리소스를 찾을 수 있습니다.

![](./images/approval-workflow/upstream-resources.png)
