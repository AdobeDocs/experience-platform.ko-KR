# ID 서비스 개요

고객의 기대에 부응하는 디지털 경험을 전달하려면 고객을 완벽하게 이해해야 합니다. 고객 데이터가 서로 다른 시스템에서 단편화되어 있는 경우 이러한 문제로 인해 각 개별 고객은 여러 개의 &quot;ID&quot;가 있는 것처럼 보입니다. Adobe Experience Platform Identity Service를 사용하면 다양한 디바이스와 시스템에 ID를 통합하여 고객 및 고객의 행동을 보다 정확하게 파악할 수 있으므로 효과적이고 개인화된 경험을 실시간으로 제공할 수 있습니다.

## ID 서비스 이해

고객은 매일 기업과 교류하고 브랜드와의 지속적인 관계를 형성합니다. 일반적인 고객은 e커머스, 충성도, 헬프데스크 시스템 등 조직의 데이터 인프라 내에서 많은 시스템에서 활성 상태일 수 있습니다. 동일한 고객은 모든 디바이스에서 익명으로 참여할 수 있습니다. Identity Service를 사용하면 고객의 전체 상황을 분석하여 서로 다른 시스템에 분산된 관련 데이터를 집계할 수 있습니다.

브랜드와의 소비자 관계에 대한 일상적인 예를 살펴보십시오.

Mary는 귀하의 전자 상거래 사이트에 과거 몇 건의 주문을 완료한 계정을 가지고 있습니다. 그녀는 보통 개인 노트북을 이용해 쇼핑하고, 매번 로그인한다. 하지만 방문 중에 태블릿을 사용하여 샌들을 구입하지만 주문을 하지 않고 로그인하지 않습니다.

이 시점에서 Mary의 활동은 두 개의 개별 프로필로 표시됩니다.디바이스 ID로 식별되는 전자 상거래 로그인 및 태블릿 디바이스

Mary는 나중에 태블릿 세션을 다시 시작하고 뉴스레터를 구독하는 동안 이메일 주소를 제공합니다. 이렇게 하면 스트리밍 인제스트는 프로필 내에서 레코드 데이터로 새로운 ID를 추가합니다. 그 결과, ID 서비스는 이제 Mary의 태블릿 장치 활동과 전자 상거래 계정 내역을 연관시킵니다.

태블릿에서 다음 번 클릭으로 타겟팅된 컨텐츠는 미확인 쇼핑객이 사용하는 태블릿이 아니라 Mary의 전체 프로필과 역사를 반영할 수 있습니다.

ID 서비스가 정의하고 유지 관리하는 ID 관계는 실시간 고객 프로파일을 통해 고객의 전체 상황을 파악하고 브랜드와의 인터랙션을 구축합니다. 자세한 내용은 실시간 [고객 프로필 개요를](../profile/home.md)참조하십시오.

### ID

ID는 엔티티(일반적으로 개인)에 고유한 데이터입니다. 로그인 ID, ECID 또는 충성도 ID와 같은 ID를 **알려진** ID라고 합니다.

이메일 주소 및 전화 번호와 같은 PII는 고객을 직접 식별합니다. 따라서 PII는 여러 시스템에서 고객의 여러 ID를 일치시키는 데 사용됩니다.

**장치를 사용하는 실제 사용자를 식별하지 않고도 알 수 없거나 익명 ID가** 분리됩니다. 이 카테고리에는 방문자의 IP 주소 및 쿠키 ID와 같은 정보가 포함됩니다. 알 수 없는 ID를 사용하여 장치에서 행동 데이터를 수집할 수 있지만, 고객이 여행 중에 PII를 제공할 때까지 장치 또는 미디어에 이러한 ID를 연결할 수 없습니다.

아래 이미지에서 보듯이, 알려진 ID와 익명 ID는 모두 [ID 그래프의](#identity-graphs)중요한 구성 요소로서 이 문서의 후반부에서 설명합니다.

![플랫폼에서 ID 연결](./images/identity-service-stitching.png)

ID 서비스 구현의 예는 다음과 같습니다.

- 텔레콤 회사는 전화 번호가 오프라인 및 온라인 데이터 세트에 모두 동일한 개인 정보를 의미하는 &quot;전화 번호&quot; 값에 의존할 수 있습니다.
- 소매 회사는 익명의 방문자의 비율이 높기 때문에 오프라인 데이터 세트에서 &quot;이메일 주소&quot;를 사용하고 온라인 데이터 세트에서 ECID를 사용할 수 있습니다.
- 은행은 지점 거래와 같은 오프라인 데이터 세트에서 &quot;계좌 번호&quot;를 선호할 수 있습니다. 대부분의 방문자가 방문 중에 인증되기 때문에 온라인 데이터 세트의 &quot;로그인 ID&quot;에 의존할 수 있습니다.
- 또한 고객은 GUID 또는 기타 범용 고유 식별자와 같은 고유한 소유권 ID를 가질 수 있습니다.

### ID 데이터

&quot;신분증이 무엇입니까?&quot;라고 물으면 더 이상의 문맥이 없다면, 그들이 유용한 답을 제공하기가 어려울 것이다. 동일한 논리로 ID가 시스템 생성 ID인지 이메일 주소인지에 관계없이 ID 값을 나타내는 문자열 값은 문자열 값 컨텍스트를 제공하는 한정자와 함께 제공되는 경우에만 완료됩니다.id 네임스페이스입니다.

## ID 네임스페이스 및 ID 그래프

고객은 온라인 채널과 오프라인 채널을 결합하여 브랜드와 인터랙션할 수 있으므로 세분화된 인터랙션을 하나의 고객 ID로 조정하는 방법을 고민할 수 있습니다.

Adobe Experience Platform은 두 가지 개념을 통해 이러한 과제를 해결해 줍니다. [ID 네임스페이스](#identity-namespaces) 및 [ID 그래프입니다](#identity-graphs).

### ID 네임스페이스

웹, 모바일 애플리케이션, 콜센터 또는 스토어를 비롯한 다양한 채널에서 고객이 브랜드와 상호 작용하는 경우 채널 전반에서 고객의 활동을 관찰하고 추적할 수 없는 경우 이를 파악하고 제공하기가 어렵습니다.

다양한 디바이스와 채널에서 고객을 이해하려면 각 채널에서 고객을 인식해야 합니다. Adobe Experience Platform은 ID 네임스페이스를 사용하여 이를 실현합니다.
ID 네임스페이스는 데이터가 시작되는 컨텍스트를 제공하는 데 사용되는 장치 ID 또는 이메일 ID와 같은 식별자입니다. ID 네임스페이스는 개별 ID를 조회하거나 연결하는 데 사용되며 ID 값에 대한 컨텍스트를 제공하여 데이터 충돌을 방지합니다. 예를 들어, ID &quot;123456&quot;은 eCommerce 시스템에서 한 사람, 도움말 데스크 시스템에서 다른 사람을 나타낼 수 있습니다. 자세한 내용은 [ID 네임스페이스 개요를](./namespaces.md)참조하십시오.

### ID 그래프

ID 그래프는 서로 다른 ID 네임스페이스 간의 관계 맵으로, 고객이 다양한 채널에서 브랜드와 상호 작용하는 방식을 시각적으로 보여줍니다.

모든 고객 ID 그래프는 고객 활동에 대한 대응으로 Identity Service에서 신속하게 통합 관리 및 업데이트됩니다.

Identity Service는 조직만 표시하고 데이터를 기반으로 하여 개인 그래프라고 하는 ID 그래프를 관리합니다. 인제스트된 데이터 레코드에 둘 이상의 ID가 포함되어 있는 경우 Identity Service가 개인 그래프를 늘리며 검색된 ID 간의 관계를 추가합니다.

ID 데이터를 제공하고 레이블을 지정할 때 고려해야 할 잠재적 유형의 예로서, &quot;직장 전화&quot;와 같은 전화 번호를 사용하면 ID 그래프에 의도한 것보다 더 많은 관계가 생길 수 있습니다. 많은 직원이 동일한 숫자를 직책에 언급하고 있으며 &quot;가정&quot; 및 &quot;모바일&quot; 서비스를 통해 관계를 최대한 정확하게 유지할 수 있습니다.

## ID 서비스에 ID 데이터 제공

이 섹션에서는 Identity Service가 각 고객에 대한 ID 그래프를 작성하기 전에 Adobe Experience Platform에 제공된 데이터를 처리하는 방법에 대해 설명합니다.

### ID 필드 결정

기업 데이터 수집 전략에 따라 ID로 레이블을 지정하는 데이터 필드는 ID 맵에 포함된 데이터를 결정합니다. Adobe Experience Platform의 이점을 최대한 활용하고 가장 포괄적인 고객 ID를 활용하려면 온라인과 오프라인 데이터를 모두 업로드해야 합니다.

- 온라인 데이터는 사용자 이름 및 이메일 주소와 같은 온라인 상태 및 행동을 설명하는 데이터입니다.

- 오프라인 데이터는 CRM 시스템의 ID와 같이 온라인 비즈니스와 직접 관련이 없는 데이터를 나타냅니다. 이러한 유형의 데이터는 ID를 더욱 견고하게 만들고 서로 다른 시스템에서 데이터 결합을 지원합니다.

### 추가 ID 네임스페이스 만들기

Experience Platform은 다양한 표준 네임스페이스를 제공하지만 ID를 적절하게 분류하려면 추가 네임스페이스를 만들어야 할 수 있습니다. 자세한 내용은 ID 네임스페이스 개요에서 조직의 [](./namespaces.md) 네임스페이스 보기 및 만들기에 대한 섹션을 참조하십시오.

>[!NOTE] ID 네임스페이스는 ID의 한정자입니다. 따라서 네임스페이스가 만들어지면 삭제할 수 없습니다.

### XDM(Experience Data Model)에 ID 데이터 포함

플랫폼이 고객 데이터를 구성하는 표준화된 프레임워크인 XDM(Experience Data Model)을 사용하면 Experience Platform과 플랫폼과 상호 작용하는 기타 서비스에서 데이터를 공유 및 이해할 수 있습니다. 자세한 내용은 XDM 시스템 [개요를](../xdm/home.md)참조하십시오.

레코드 및 시간 시리즈 스키마 모두 ID 데이터를 포함할 방법을 제공합니다. 데이터를 인제스트하면 ID 그래프가 공통 ID 데이터를 공유하는 경우 서로 다른 네임스페이스의 데이터 조각 간에 새로운 관계를 만듭니다.

### XDM 필드를 ID로 표시

레코드 또는 시간 시리즈 XDM 클래스를 구현하는 `string` 스키마의 모든 유형의 필드는 ID 필드로 레이블이 지정될 수 있습니다. 따라서 해당 필드에 인제스트된 모든 데이터는 ID 데이터로 간주됩니다.

ID 필드는 ID가 공통 PII 데이터를 공유하는 경우 ID를 연결할 수도 있습니다.
예를 들어, ID 서비스는 전화 번호 필드에 ID 필드에 레이블을 지정함으로써 동일한 전화 번호를 사용하고 있는 것으로 확인된 다른 개인과의 관계를 자동으로 그래프로 표시합니다.

>[!NOTE] 결과 ID의 명칭 공간은 필드의 레이블이 지정된 시점에 제공됩니다.

### ID 서비스에 대한 데이터 집합 구성

스트리밍 통합 프로세스 동안 Identity Service는 레코드 및 시간 시리즈 데이터에서 ID 데이터를 자동으로 추출합니다. 하지만 데이터를 인제스트하려면 Identity Service에 대해 활성화되어야 합니다. 자세한 내용은 API를 사용하여 실시간 고객 프로파일 및 ID 서비스용 데이터 세트 [구성에 대한](../profile/tutorials/dataset-configuration.md) 자습서를 참조하십시오.

### 데이터를 ID 서비스로 인제스트

Identity Service는 [일괄 처리](../ingestion/batch-ingestion/overview.md) 또는 [스트리밍 인제스트를](../ingestion/streaming-ingestion/overview.md)통해 Experience Platform으로 전송된 XDM 준수 데이터를 사용합니다.

## 데이터 거버넌스

Adobe Experience Platform은 개인 정보를 고려하여 구축되었으며 고객 PII 데이터를 보호하기 위한 데이터 거버넌스 프레임워크를 포함하고 있습니다. &quot;email&quot; 또는 &quot;phone&quot; 네임스페이스에 있는 ID 데이터는 기본적으로 암호화되지만 중요한 데이터가 지속되기 전에 암호화되도록 하기 위해 데이터 사용 레이블을 인제스트되거나 플랫폼에 도착하면 데이터에 적용할 수 있습니다. 자세한 내용은 데이터 거버넌스 [개요를](../data-governance/home.md)참조하십시오.
